## 1. 目标

把 BioSpark-Light（FastAPI + PyTorch 桌面版生物信号实验台）做成 **终端用户无需安装 Python 也能直接双击运行的可执行文件**，并把整个源码仓库公开发布到 GitHub。

2. `dist/BioSpark-Light/BioSpark-Light.exe` 在干净环境（无 Python）双击即可启动。

## 2. 关键决策与理由

### 2.1 PyInstaller 一文件夹（one-folder） vs 单文件（one-file）

选择**一文件夹模式**。理由：

| 维度 | one-folder ✅ | one-file ❌ |
|------|--------------|------------|
| Torch DLL 兼容性 | 直接运行，DLL 就在 `_internal/torch/lib/` | 启动时解压到 `%TEMP%\_MEIxxxxx\`，部分 torch 的 `.pyd` 在解压后路径里加载失败（GitHub 上有大量同类 issue） |
| 冷启动 | 跳过解压，2-3 秒进入应用 | 每次启动都要解压 ~700MB 到临时目录，10-30 秒 |
| 杀软误报 | 较低 | 极高（自解压可执行天然像 dropper） |
| 排错 | 可手工进 `_internal/` 看缺哪个文件 | 黑盒，出问题只能整个重打包 |
| 分发 | 整个文件夹打 zip | 单 .exe |

唯一代价：分发时是一个文件夹（用户解压 zip 后双击里面的 .exe），不是孤立的 .exe。这点完全可以接受。

### 2.2 CPU-only torch wheel

`pip install torch --index-url https://download.pytorch.org/whl/cpu`

CUDA wheel 会让最终 bundle 涨到 ~3.5 GB（多出来全是 CUDA runtime DLL），而 BioSpark-Light 是本地 inference-free 的轻量训练，CPU 完全够用。最终 bundle ~747 MB。

### 2.3 `console=True`（v0.1）

故意保留控制台窗口，让用户能看到启动日志。第一版用户少、问题多，启动黑屏 = 排错地狱。`docs/BUILD.md` 里写了一行改 `console=False` 的指引，等 UI 稳定再切换。

### 2.4 不使用 UPX 压缩

虽然 UPX 能把 bundle 压到 ~450 MB，但 UPX 经常破坏 numpy / torch 的 native DLL（已有大量 GitHub issue）。空间换稳定，不开 UPX。

### 2.5 frozen-mode 路径解析

打包后 `__file__` 不可信，`backend/config.py` 已加入：

```python
if getattr(sys, "frozen", False) and hasattr(sys, "_MEIPASS"):
    BASE_DIR = Path(sys._MEIPASS)  # PyInstaller 提供的运行时根
else:
    BASE_DIR = Path(__file__).resolve().parent.parent  # 源码运行
FRONTEND_DIR = BASE_DIR / "frontend"
```

这样源码模式（`python launcher.py`）和打包模式（`BioSpark-Light.exe`）共用同一份代码，FastAPI 的 StaticFiles 永远指对地方。

### 2.6 用户数据 vs 程序数据严格分离

- **只读、随程序走**：`frontend/` 静态文件 → 通过 `app_datas` 打进 `_internal/frontend/`，运行时从 `sys._MEIPASS / "frontend"` 加载。
- **可写、随用户走**：DB / uploads / checkpoints → `platformdirs.user_data_dir(APP_NAME, appauthor=False)` → `%LOCALAPPDATA%\BioSpark-Light\`。

好处：用户卸载/重装 .exe 不丢历史训练记录；多用户在同一台机器上 home dir 隔离；安装到 `Program Files` 也不会因写权限不足崩溃。

## 3. 关键产出物

```
biospark-light.spec      ← PyInstaller 入口 spec（手工调优）
backend/config.py        ← 加入 frozen-mode 检测
docs/BUILD.md            ← 用户构建指南
docs/R&D-REPORT-...md    ← 本文档
.gitignore               ← 给 biospark-light.spec 开白名单
```

`biospark-light.spec` 关键点：

- `collect_all("torch" / "sklearn" / "scipy" / "matplotlib")` — 让 PyInstaller 抓全这四个重型框架的 binaries / datas / hidden imports。
- `app_datas = [("frontend", "frontend")]` — 把整个 frontend 目录作为数据塞进 bundle。
- 显式 `extra_hidden` 列表 25 项，覆盖 uvicorn / SQLAlchemy / pydantic / 项目自身 backend.* 模块——PyInstaller 静态分析常漏的就是动态导入和子模块。
- `excludes` 14 项：tkinter / Qt / Jupyter / pytest / torchvision / torchaudio / mne 等本项目用不到的大块头，能从 bundle 切掉 ~150 MB。

# 附录 A — v0.1.1 修复轮次：数据泄漏 + OpenBCI 接入

**日期**：2026-04-29
**触发事件**：用户用 5 类手势(CLENCH / FIVE / OK / ROCK / TWO)训练后,混淆矩阵全是 100%,t-SNE 五个簇分得"过分干净",疑似过拟合或数据泄漏。

## A.1 诊断:为什么会 100%

旧代码三个独立缺陷叠加,使得"100% on val"是预期内的退化结果,不是 bug:

1. **窗级随机切分** — `trainer._run_training` 用 `np.random.permutation(n_samples)` 直接对每个窗口洗牌切 80/20。同一段录音切出的相邻窗会被分到 train 和 val 两边,EMG 在同一 trial 内极度相关,等于让模型背答案。
2. **分段时丢失了来源信息** — `data_preparator.segment_*` 输出 `(segment, label)` 平表,没记录每个 segment 来自哪段原始录音 → 下游做 group-aware 切分都做不了。
3. **混淆矩阵跑在 val 上,且没有独立 test** — `compute_confusion_matrix` 在 val 上报数,而 val 又是早停看的那份,等于双重 peeking。

50k 参数的 1D-CNN 在 ~250 个训练窗上完全够"背"出对应关系,所以非对角线全是精确的 0。

## A.2 修复(分层落地,可独立回滚)

### Tier 1:消灭窗级泄漏(核心修复)

| 文件 | 改动 |
|------|------|
| `backend/services/data_preparator.py` | 三种 Mode 全部输出 `__group__` 列(Mode A:文件路径;Mode B:interval 索引;Mode C:文件名) |
| `backend/services/dataset_loader.py` | 识别 `__group__` 列并从信号列里剔除;在 summary 里暴露 `group_column` / `n_groups` |
| `backend/services/trainer.py` | 新增 `_split_indices()`,用 `sklearn.model_selection.GroupShuffleSplit` 做组级三路切分(train/val/test);旧 CSV 没有 `__group__` 时降级为逐行随机(行为与旧代码一致) |

合成数据验证:5 类 × 4 录音/类 × 10 重叠窗 = 200 个 segment,共 20 个 group。组级切分后 train/val/test 三个 group 集合**两两不相交**(12/4/4 个 group),证明同一录音的窗不会跨切分泄漏。

### Tier 2:让"诚实评估"成为默认

| 改动 | 说明 |
|------|------|
| 三路切分 | 默认 `train:val:test = 0.6:0.2:0.2`,test 留作最终评估,val 只用于早停。`compute_confusion_matrix` 优先在 test 上跑 |
| 泄漏 banner | 当 `accuracy ≥ 0.99 且任一类支持数 < 30` 时,API 在 `/confusion_matrix` 响应里返回 `leakage_warning` 对象,前端在结果页顶部打黄色警告 |
| 训练前 banner | 训练标签页加载 CSV 时,若没有 `__group__` 列就在数据预览下方显示黄色 banner,提醒用户回 Prep 重新切分 |
| WS `start` 事件多带 `split` | 训练开始的第一行 epoch 日志里就打印"组级切分 · N 组"或"逐行随机切分(缺少 __group__ 列)",绿/黄区分,让用户立刻看到自己掉进哪个分支 |
| Overlap 提示 | Prep 标签页 `Overlap Ratio` 字段下方,只有 > 0 才显示警告——重叠窗会人为膨胀样本数 |

### Tier 3:OpenBCI / 真实采集设备接入

用户实际采集流程是 **OpenBCI Cyton + .txt 输出**,典型协议是「5 秒一段 × 60 段,每段前 1 秒为有效任务窗口」。旧 Prep 完全无法处理:`.txt` 不收、`%` 注释行不跳、5 列通道选不了、间隔截取不会。补齐:

| 能力 | 实现 |
|------|------|
| `.txt` 文件 | `data_preparator._read_csv_or_txt()` 跳 `%` 注释行后用 pandas 解析。Mode A/B/C 的 zip 遍历放宽到 `(.csv, .txt)` |
| OpenBCI 头部识别 | `_parse_openbci_header()` 正则匹配 `%Sample Rate = 250 Hz`,inspect 接口自动建议采样率 |
| 多通道列选择 | `signal_col_indices: list[int]` 覆盖原来的单列 `signal_col_index`。多通道时输出列名变成 `ch{c}_{i}`,与现有 `dataset_loader._detect_channel_structure` 的正则匹配,trainer 自然 reshape 成 `(N, C, L)` |
| 间歇采样 stride | `_segment(stride=...)` 接受样本数级别的步长。`segment_length=1s + stride=5s` 直接表达"每 5s 取前 1s" |
| 列范围自动建议 | `_guess_signal_columns()` 优先识别 `EXG Channel N` 命名模式,否则取最长连续数值列段。OpenBCI 8 通道默认建议 `[1..5]` 或 `[1..8]` |
| `group_by` 选项 | `recording`(默认)= 一文件一组;`trial` = 每段都自成一组。一类只有 1 段长录音 + 60 个独立 trial 的场景,选 `trial` 才能让 train/val/test 都各包含全部 5 类 |

**前端**:Prep 标签页加 3 个新字段:
- `Multi-channel Columns` 文本框,接受 `1-5` 或 `1,2,3,4,5`,inspect 检测到 OpenBCI 时自动填好
- `Stride (sec, optional)` 数字框,留空时按 overlap 比例切分
- `Group-aware split unit` 下拉(Recording / Trial)

### A.3 端到端验证(合成 OpenBCI 数据)

按用户实际场景造数据:5 类 × 1 个 .txt × 5 min × 250 Hz × 5 通道 + `%`-头,Cyton 命名风格:

```
inspect:
  openbci_detected = True
  suggested_sampling_rate = 250.0   ← 从 %Sample Rate 头自动读出
  suggested_signal_col_indices = [1, 2, 3, 4, 5]   ← EXG Channel 0..4

segment_long_recordings(seg=1s, stride=5s, indices=[1..5], group_by=trial):
  shape = (300, 1252)               ← 5 类 × 60 trial = 300,5 通道 × 250 + label + __group__
  per-class segments = 60/60/60/60/60  ← 5min/5s = 60,完全对齐协议
  unique groups = 300                  ← trial 级
  channel_detected = True, n_channels = 5, signal_length = 250

_split_indices(group_by=trial):
  train=180  val=60  test=60
  train classes = {ALL 5}  val classes = {ALL 5}  test classes = {ALL 5}   ← 关键:每份切分都见过所有类
```

对照组(`group_by=recording`):一类只有 1 个 group → GroupShuffleSplit 把整个 class 扔到某一份切分里,test 集只有一类 → 混淆矩阵 4 行全空。这正是为什么协议结构清楚(每段含多个独立 trial)时**必须**选 `trial` 级。

### A.4 文件改动清单

```
backend/services/preprocess.py        ← _segment 加 stride 参数,支持 1D/2D 信号
backend/services/data_preparator.py   ← 主战场:.txt + 多通道 + stride + group_by + OpenBCI 头识别
backend/services/dataset_loader.py    ← 识别 __group__ 列,在 summary 暴露
backend/services/trainer.py           ← _split_indices 三路 group-aware 切分;compute_confusion_matrix 跑 test

backend/routers/training.py           ← TrainStartRequest 加 test_split,持久化 test 集到 TrainingJob
frontend/index.html                   ← 3 个新 prep 字段
frontend/js/prep.js                   ← 列范围解析 + 字段绑定 + inspect 自动填

README.md                             ← 加"务必走 Prep 标签页"的 ⚠️ 提示段
```

## A.5 已知限制(交给 v0.2)

| 限制 | 说明 |
|------|------|
| 一段录音内的 trial 仍有时间相关性 | 即使 `group_by=trial`,同一长录音里相邻 trial 的肌张力/电极漂移仍相似。**真正的泛化评估**需要同受试者跨 session 多次录音,甚至跨受试者 LeaveOneSubjectOut |
| 没有 GroupKFold 交叉验证 | 当前是单次 80/20/20 切分。要发表级别的 mean ± std macro-F1,得加 `cv_folds` 配置走 `sklearn.GroupKFold`(列在 v0.2 路线图) |
| `group_by=trial` 时降级为逐行随机 | `n_groups == n_samples` 触发 `_split_indices` 的 degenerate 分支,泄漏 banner 仍会 fire(因为 `group_aware=False`) — 这是有意为之的诚实信号,不是 bug |
| 没有 temporal split | 一些时序协议希望"前 80% 时间段训练 / 后 20% 测试",目前没有这个选项 |
| 单受试者假设 | 数据 schema 里没有 `subject_id` 字段。多受试者上线后要新增 |

# 附录 B — v0.2 发布:1D-CNN + Transformer 混合架构 + 自动推荐

**日期**:2026-05-03
**版本**:v0.2.0-light
**触发动机**:用户(研究生面试场景,EEG/EMG 多通道分析,论文导向)反馈 v0.1 的纯 1D-CNN 在长序列多通道数据上抓不到长程依赖,且消融实验缺一个对照架构;另一方面小样本上盲目上 Transformer 又必过拟合 → 需要"两个架构 + 系统先推荐 + 用户可覆盖"的设计。

## B.1 设计:为什么是混合架构而不是纯 Transformer

候选方案对比:

| 方案 | 参数量 | 小样本表现 | 论文价值 | 否决理由 |
|------|--------|-----------|---------|---------|
| 纯 Transformer1D | 500K-2M | 极差(必过拟合) | 低(已被 ViT/AST 覆盖) | 不适合本工具的小样本生理信号场景 |
| 全卷积 ResNet1D | 100-300K | 好但已是同类 | 中 | 与现有 1D-CNN 同质化,加它不解决长程依赖问题 |
| **CNN + Transformer 混合** ✅ | 100-400K | 中等数据可用 | 高(消融对比 CNN vs Hybrid) | — |

**关键架构选择**:
- Conv stem(局部特征 + 下采样 4x)→ Patch embed(1×1 Conv 升维到 d_model=128)→ 可学习位置编码 → Transformer encoder(2 层 × 4 头 × FFN 256)→ Mean pool → Classifier
- d_model=128 + 2 层是参数量与表达力的甜区:实测 ~358K 参数(原计划 100-300K,实际略超),在小数据上比纯 Transformer 稳健
- `norm_first=True` + GELU,与 ViT/Pre-LN Transformer 主流配置对齐
- 长序列(>max_tokens=512)由 `_to_tokens()` 截断而非滑窗,简单且对生理信号场景够用

## B.2 推荐函数:5 条规则,可读理由

`backend/services/auto_optimizer.py::recommend_architecture` 输入 `(n_samples, n_channels, signal_length, n_groups)`,输出 `{recommended, confidence, reason_zh, reason_en, alternatives}`。

规则按优先级:

| # | 条件 | 推荐 | 置信度 | 理由 |
|---|------|------|--------|------|
| 1 | `n_samples < 500` 或 `n_groups < 5` | cnn | 0.85 | 数据/被试太少,混合模型易过拟合 |
| 2 | `signal_length < 128` | cnn | 0.85 | 序列太短,注意力没什么可建模的长程依赖 |
| 3 | `n_samples ≥ 500 且 signal_length ≥ 256 且 n_channels ≥ 2` | hybrid | 0.85 | 数据充足 + 长序列 + 多通道,注意力收益最大 |
| 4 | `n_samples ≥ 2000` | hybrid | 0.75 | 数据足够支撑更高容量模型 |
| 5 | 其他(中间区) | cnn | 0.65 | 保守选择,先跑基线 |

每条规则都附中英双语 reason 字符串和 alternatives 数组(每个含 `tradeoff_zh/en`),前端直接渲染给用户看,这是"系统推荐 + 用户可决定"流程的关键——不是黑箱推荐,是有理由可质疑的推荐。

## B.3 双模式 UX:推荐 vs 用户覆盖

`TrainStartRequest.arch` 三选一:

| 值 | 行为 | 训练记录 `user_overrode_recommendation` |
|----|------|------------------------------------------|
| `auto`(默认)| 后端调 `recommend_architecture` 决定 | `False` |
| `cnn` 且推荐也是 cnn | 与推荐一致 | `False` |
| `hybrid` 且推荐也是 hybrid | 与推荐一致 | `False` |
| `cnn` 但推荐是 hybrid | **用户覆盖** | `True` |
| `hybrid` 但推荐是 cnn | **用户覆盖** | `True` |

WS `start` payload 多带 `model_arch` / `arch_recommendation` / `user_overrode_recommendation`,前端结果页能显示"已覆盖系统推荐"标记 → 论文里做消融时能直接读出哪些 run 是用户主动选的对照。

前端"模型选择"区块布局:
- **数据画像**(profile 卡片):n_samples / n_channels / signal_length / n_groups / n_classes — 让用户能自评
- **推荐卡片**:高亮显示系统推荐架构 + 中文/英文理由 + 置信度条
- **三选项单选**:`[● 自动(推荐 X)] [○ 1D-CNN] [○ Hybrid]`,每项展示参数量 + 训练时间相对量级 + 一句适用场景
- **覆盖警告**:切到非推荐项时显示黄色提示"已覆盖系统推荐"

实现要点:`_renderArchRecommendation` 缓存 assess data,切换架构时纯前端 re-render(无需重新调 assess 接口),响应即时。

## B.4 跨架构 warm-start 兼容

旧 v0.1 的 warm-start 链是按 `state_dict` 形状对齐的——CNN ↔ Hybrid 的 state_dict 完全不重合,所以切架构时:
- 现有的"按 shape 过滤,不匹配的层 skipped"逻辑天然处理:cnn warm-start 加载到 hybrid 时所有层 skip,等价于从头训练
- 不需要新代码,但要在文档里点明:"切架构 + warm-start = 等价于不 warm-start",避免用户期望落空

## B.7 已知限制(交给 v0.3)

| 限制 | 说明 |
|------|------|
| 注意力可视化未实现 | Hybrid 的 attention weights 已缓存,但前端没渲染。v0.3 配 Grad-CAM 一起做"模型在看哪里"页 |
| Hybrid 在 1 通道短序列上常被推荐 cnn | 这是规则的本意,不是限制——但用户如果坚持想试 hybrid,需要看到"训练时间长 ~2x"的代价提示(已实现) |
| 推荐函数没考虑类不平衡度 | 极端不平衡(如 10:1)对架构选择的影响未纳入规则 |
| 没有论文级对比实验 | 同一数据上 cnn vs hybrid 的 mean ± std macro-F1 差异未量化,需要配 GroupKFold(v0.3 路线) |
| max_tokens=512 是硬截断 | 超长生理信号(如 30s × 250Hz = 7500 点)会丢尾部信息。下一版加 stride/overlap patch 化 |
